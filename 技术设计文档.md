## 1. 项目概述

《六合》是一款基于六边形网格的双人对弈棋盘游戏，核心机制包含：落子、吃子（围杀）、圈地（封闭领地）三个阶段，游戏结束时以存活棋子数 + 领地格数计算总分。

**设计原则：**
- 逻辑层与渲染层完全分离
- 数据驱动，状态单一来源（Single Source of Truth）
- 使用 Godot 信号系统解耦模块通信

## 4. 核心数据结构

### 4.1 坐标系：`HexCoord`

采用 **Cube 坐标系**。每个六边形由三个整数 `(q, r, s)` 标识，满足约束：

```
q + r + s = 0
```

因此实际只需存储 `(q, r)`，`s` 可由 `s = -q - r` 推导。

```
HexCoord
├── q: int
├── r: int
├── s: int  (只读，由 q 和 r 计算)
├── DIRECTIONS: Array[HexCoord]  # 六个方向的单位向量（常量）
│     # 按顺序：右、右上、左上、左、左下、右下
├── add(other: HexCoord) -> HexCoord
├── equals(other: HexCoord) -> bool
├── neighbors() -> Array[HexCoord]     # 返回六个相邻坐标
├── distance(other: HexCoord) -> int   # Cube 距离公式
└── to_string() -> String              # 用于 Dictionary 键
```

**六个方向向量（平顶六边形）：**

```
方向索引  (q,  r)
   0      +1,  0   右
   1      +1, -1   右上
   2       0, -1   左上
   3      -1,  0   左
   4      -1, +1   左下
   5       0, +1   右下
```

**Cube 坐标 → 像素坐标（平顶六边形）：**

```
hex_size = 格子的外接圆半径（像素）

pixel_x = hex_size * (3/2 * q)
pixel_y = hex_size * (√3/2 * q + √3 * r)
```

**像素坐标 → Cube 坐标（点击判断）：**

```
q_frac = (2/3 * x) / hex_size
r_frac = (-1/3 * x + √3/3 * y) / hex_size
# 再对 (q_frac, r_frac) 做 Cube Round 取整
```

**Cube Round 取整算法（必须实现，否则边界点击会出错）：**

```
将 (q_frac, r_frac) 四舍五入为整数 (rq, rr, rs)
计算各分量的差值 dq, dr, ds
将差值最大的分量用另外两个重新计算
```

### 4.2 单元格状态：`CellState`

```
enum CellState {
    EMPTY,           # 空格
    BLACK,           # 黑方棋子
    WHITE,           # 白方棋子
    TERRITORY_BLACK, # 黑方永久领地
    TERRITORY_WHITE  # 白方永久领地
}
```

### 4.3 棋盘数据：`HexBoard`

```
HexBoard
├── cells: Dictionary          # { "q,r" -> CellState }
├── board_radius: int          # 棋盘半径（从中心到边缘的格子数）
├── all_coords: Array[HexCoord]  # 棋盘内所有合法坐标（初始化时计算）
│
├── initialize(radius: int)    # 生成棋盘，填充所有坐标为 EMPTY
├── is_valid_coord(coord) -> bool   # 坐标是否在棋盘内
├── get_cell(coord) -> CellState
├── set_cell(coord, state: CellState)
├── get_neighbors(coord) -> Array[HexCoord]  # 仅返回棋盘内的相邻格
├── get_empty_cells() -> Array[HexCoord]
├── is_territory(coord) -> bool
└── clone() -> HexBoard        # 深拷贝，用于历史记录/悔棋
```

**棋盘生成算法（半径为 N 的六边形棋盘）：**

```
所有满足 max(|q|, |r|, |s|) <= N 的坐标构成棋盘
共计 3N² + 3N + 1 个格子
```

### 4.4 游戏状态：`GameState`

```
enum Player { BLACK, WHITE }
enum Phase { WAITING, PLACING, RESOLVING_CAPTURE, RESOLVING_TERRITORY, GAME_OVER }

GameState
├── board: HexBoard
├── current_player: Player
├── phase: Phase
├── consecutive_passes: int     # 连续 Pass 计数，达到 2 则游戏结束
├── move_history: Array         # 每步操作记录（用于悔棋）
├── scores: Dictionary          # { Player -> int }
│
├── switch_player()
├── record_pass()
├── can_place_at(coord) -> bool  # 综合合法性检查
└── is_game_over() -> bool
```

## 5. 六边形网格系统

### 5.1 坐标转换模块职责

`HexCoord.gd` 是纯数学模块，不依赖任何 Godot API，设计为 **静态工具类**，便于在逻辑层随时调用。

### 5.2 棋盘合法坐标集合

在 `HexBoard.initialize()` 时，一次性生成所有合法坐标并缓存到 `all_coords`，后续所有算法直接查询此集合，避免重复计算。

### 5.3 邻格查询

```
get_neighbors(coord):
    result = []
    for direction in HexCoord.DIRECTIONS:
        neighbor = coord.add(direction)
        if is_valid_coord(neighbor):
            result.append(neighbor)
    return result
```

这是吃子算法和领地算法的基础原语，必须正确处理边界（棋盘边缘的格子邻格数少于 6）。

## 6. 游戏状态机

### 6.1 状态转换图

```
          ┌─────────────────────────────────────┐
          │                                     │
    [WAITING] ──玩家点击格子──→ [PLACING]         │
          ↑                        │            │
          │              落子成功   │            │
          │                        ↓            │
          │              [RESOLVING_CAPTURE]     │
          │                        │            │
          │              吃子结算完毕│            │
          │                        ↓            │
          │           [RESOLVING_TERRITORY]      │
          │                        │            │
          │              领地结算完毕│            │
          │                        ↓            │
          └──────────切换玩家──── [WAITING]      │
                                               │
          [WAITING] ──玩家 Pass──→ 记录 Pass     │
                                  │            │
                         连续2次Pass│            │
                                  ↓            │
                            [GAME_OVER] ────────┘
                         棋盘无合法落点也触发
```

### 6.2 落子合法性检查 `can_place_at(coord)`

按顺序检查以下条件，任一不满足则返回 `false`：

1. `coord` 在棋盘范围内
2. `board.get_cell(coord) == EMPTY`（格子为空）
3. `board.get_cell(coord)` 不是任何一方的永久领地（`TERRITORY_BLACK` 或 `TERRITORY_WHITE`）

### 6.3 回合执行流程

```
execute_turn(coord):
    1. 验证 can_place_at(coord)，不合法则返回
    2. board.set_cell(coord, current_player's piece)
    3. captured = CaptureResolver.resolve(board, current_player)
    4. 从棋盘移除 captured 中的所有坐标
    5. new_territory = TerritoryResolver.resolve(board, current_player)
    6. 将 new_territory 标记为当前玩家的领地
    7. consecutive_passes = 0
    8. record_move(coord, captured, new_territory)
    9. switch_player()
    10. 发出信号 turn_completed
    11. 检查游戏是否结束
```

## 7. 核心算法

### 7.1 连通体查找（BFS 洪水填充）

这是吃子和领地算法共用的基础算法。

```
find_group(board, start_coord, target_state) -> Array[HexCoord]:
    """
    从 start_coord 出发，找到所有与其相连且状态为 target_state 的格子。
    """
    if board.get_cell(start_coord) != target_state:
        return []

    visited = {}
    queue = [start_coord]
    group = []

    while queue 不为空:
        current = queue.pop_front()
        if current in visited: continue
        visited[current] = true
        group.append(current)

        for neighbor in board.get_neighbors(current):
            if neighbor not in visited:
                if board.get_cell(neighbor) == target_state:
                    queue.append(neighbor)

    return group
```

### 7.2 吃子算法 `CaptureResolver`

**核心概念：气（Liberty）**

一个棋子群的"气"是其所有相邻的空格数量（不含被己方棋子或棋盘边界占据的格子）。气为 0 时，该群被吃掉。

```
resolve(board, attacker: Player) -> Array[HexCoord]:
    """
    检查所有敌方棋子群，返回所有需要被移除的坐标。
    attacker 是刚刚落子的玩家。
    """
    defender_state = 对手的棋子状态（BLACK 或 WHITE）
    captured = []
    visited_groups = {}  # 避免重复检查同一群

    for coord in board.all_coords:
        if board.get_cell(coord) != defender_state: continue
        if coord in visited_groups: continue

        group = find_group(board, coord, defender_state)
        for c in group: visited_groups[c] = true

        if get_liberties(board, group) == 0:
            captured.extend(group)

    return captured

get_liberties(board, group: Array) -> int:
    """计算棋子群的气数。"""
    liberty_set = {}
    for coord in group:
        for neighbor in board.get_neighbors(coord):
            state = board.get_cell(neighbor)
            if state == EMPTY:
                liberty_set[neighbor] = true
    return liberty_set.size()
```

**重要边界情况：**

- **自杀手判断：** 当前规则文档未提及禁止自杀，但如果需要实现，可在落子后检查己方棋子群的气是否为 0（先吃子后检查自身）。
- **落子顺序：** 先检查敌方是否被吃，再检查自身。即：一手棋如果同时导致己方和敌方气为 0，敌方先被移除，己方可能因此获得气而存活。

### 7.3 领地算法 `TerritoryResolver`

**核心思路：** 一片空格区域构成某方的领地，当且仅当：
1. 该区域与棋盘边界**不相连**（即完全被棋子或边界围住）
2. 区域内**没有**敌方棋子
3. 包围该区域的棋子**全部**属于同一方玩家

```
resolve(board, placer: Player) -> Array[HexCoord]:
    """
    检查所有空格区域，返回新增的该玩家领地坐标。
    """
    new_territory = []
    visited = {}

    for coord in board.all_coords:
        state = board.get_cell(coord)
        if state != EMPTY: continue
        if coord in visited: continue

        # 对每片连通空格区域做分析
        region = find_group(board, coord, EMPTY)
        for c in region: visited[c] = true

        owner = determine_region_owner(board, region)
        if owner == placer:
            new_territory.extend(region)

    return new_territory

determine_region_owner(board, region: Array) -> Player or null:
    """
    分析围绕此空格区域的棋子属于哪一方。
    如果：
      - 区域接触棋盘边界 → 不是领地，返回 null
      - 区域内部有任意敌方棋子 → 不是领地（规则要求），返回 null
      - 围绕棋子属于多方 → 中立，返回 null
      - 仅被一方棋子围绕 → 返回该方
    """
    border_players = {}  # 记录接触到的棋子颜色
    touches_boundary = false

    for coord in region:
        neighbors = board.get_neighbors(coord)

        # 检查是否接触边界（邻格数 < 6 说明在棋盘边缘）
        if neighbors.size() < 6:
            touches_boundary = true
            break

        for neighbor in neighbors:
            state = board.get_cell(neighbor)
            if state == BLACK: border_players[BLACK] = true
            elif state == WHITE: border_players[WHITE] = true

    if touches_boundary: return null
    if border_players.size() != 1: return null  # 多方或无人围绕

    return border_players.keys()[0]
```

**关于"接触边界"的判断说明：**

一个坐标是否在棋盘边缘，可通过检查其邻格数量是否小于 6 来判断（边缘格子的部分方向在棋盘外）。这比计算 Cube 距离更通用，且自动适应不同形状的棋盘。

更严格的实现：在 `find_group` 的洪水填充中，若任意一步扩展时发现相邻方向超出棋盘边界，则标记 `touches_boundary = true`。

### 7.4 计分算法 `ScoreCalculator`

```
calculate(board) -> Dictionary:
    result = { BLACK: 0, WHITE: 0 }

    for coord in board.all_coords:
        state = board.get_cell(coord)
        match state:
            BLACK:           result[BLACK] += 1
            WHITE:           result[WHITE] += 1
            TERRITORY_BLACK: result[BLACK] += 1
            TERRITORY_WHITE: result[WHITE] += 1

    return result
```

## 8. 渲染系统

### 8.1 渲染层架构

渲染层只监听信号，不持有任何游戏逻辑状态。所有渲染决策基于从 `GameState` 读取的数据，保证数据单一来源。

```
渲染层结构：

BoardRenderer（棋盘渲染器）
├── 负责：绘制六边形格子、边界、坐标网格
├── 监听：board_initialized 信号
└── 持有：HexCell 节点池（对象池优化）

PieceRenderer（棋子渲染器）
├── 负责：添加/移除棋子节点、落子动画
├── 监听：piece_placed, pieces_captured 信号
└── 持有：Piece 节点字典 { coord_key -> PieceNode }

TerritoryRenderer（领地渲染器）
├── 负责：绘制领地半透明遮罩
├── 监听：territory_formed 信号
└── 使用：带透明度的 Polygon2D 叠加
```

### 8.2 HexCell 节点结构（预制件）

```
HexCell (Node2D)
├── Background (Polygon2D)     # 六边形底色
├── Border (Line2D / Polygon2D) # 边框
├── Highlight (Polygon2D)       # 鼠标悬停高亮，初始不可见
└── script: HexCellView.gd
```

六边形顶点计算（平顶，外接圆半径 R）：

```
对 i in 0..5:
    angle = 60° * i  (平顶从 0° 开始)
    vertex_x = R * cos(angle)
    vertex_y = R * sin(angle)
```

### 8.3 棋子节点结构（预制件）

```
Piece (Node2D)
├── Circle (Polygon2D 或 Sprite2D)  # 棋子外形
├── Shadow (Polygon2D)               # 可选阴影效果
└── script: PieceView.gd

PieceView
├── set_color(player: Player)
├── play_place_animation()   # 落子时的缩放动画（Tween）
└── play_capture_animation() # 被吃时的淡出动画（Tween）
```

### 8.4 坐标系对齐

渲染层持有一个 `HexLayout` 对象，统一管理：

```
HexLayout
├── hex_size: float          # 格子外接圆半径（像素）
├── origin: Vector2          # 棋盘中心在屏幕上的像素坐标
├── cube_to_pixel(coord) -> Vector2
└── pixel_to_cube(pixel: Vector2) -> HexCoord
```

所有渲染节点的位置通过 `cube_to_pixel` 计算，确保与点击检测使用同一坐标系，不会出现偏移。

- 若格子合法（可落子）：显示绿色半透明高亮
- 若格子不合法（已有棋子/领地）：显示红色半透明高亮或不显示
- 高亮效果在 `HexCell.Highlight` 节点上切换 `visible` 和 `modulate`

在 `_input` 中同时处理 `InputEventScreenTouch`，逻辑与鼠标点击相同，便于移植到移动端。

信号是本项目的核心通信机制。所有跨层通信必须通过信号，禁止渲染层直接调用逻辑层方法（单向依赖）。

### 11.1 信号定义总表

| 信号名 | 发出者 | 携带参数 | 监听者 |
|--------|--------|----------|--------|
| `board_initialized` | GameState | `board: HexBoard` | BoardRenderer |
| `cell_clicked` | InputHandler | `coord: HexCoord` | GameState |
| `cell_hovered` | InputHandler | `coord: HexCoord` | BoardRenderer（高亮） |

## 12. 扩展功能

### 12.1 悔棋（Undo）

在 `GameState.move_history` 中，每步记录：

```
MoveRecord {
    coord: HexCoord         # 落子坐标
    player: Player          # 执行者
    captured: Array         # 被吃坐标列表
    territory: Array        # 新增领地坐标列表
}
```

悔棋时，从 history 弹出最后一条记录，逆向还原棋盘状态（移除落下的子、恢复被吃的子、撤销领地标记）。

### 12.2 AI 对手

**实现层级（按难度递增）：**

| 级别 | 策略 | 说明 |
|------|------|------|
| 简单 | 随机落子 | 从合法格中随机选择 |
| 中等 | 贪心算法 | 优先选择能吃子或能圈地的位置 |
| 困难 | Minimax + Alpha-Beta 剪枝 | 搜索深度 3-4，需要评估函数 |
| 专家 | MCTS（蒙特卡洛树搜索）| 模拟对弈，无需手工评估函数 |

**AI 接口设计：**

```
BaseAI (抽象类)
└── choose_move(board, player) -> HexCoord or null
```

将 AI 模块挂载为 `GameState` 的观察者，当轮到 AI 玩家时，自动调用 `choose_move` 并执行落子，无需改动核心逻辑。

### 12.3 联机对战

使用 Godot 内置 **MultiplayerAPI**（ENet / WebSocket）：

- 服务端持有唯一的 `GameState`，客户端只做渲染
- 落子操作通过 RPC（Remote Procedure Call）传递
- 需额外实现：大厅匹配、连接断开处理、回合超时

### 12.4 存档与读档

使用 Godot 的 `FileAccess` API，将 `HexBoard.cells` 序列化为 JSON 存入本地文件，读档时还原 `GameState`。

## 13. 性能与边界情况

### 13.1 性能分析

| 操作 | 时间复杂度 | 说明 |
|------|-----------|------|
| 邻格查询 | O(1) | 固定 6 个方向 |
| 连通体查找 | O(V) | V = 格子总数 |
| 吃子检查 | O(V) | 遍历所有敌方格子一次 |
| 领地检查 | O(V) | 遍历所有空格一次 |
| 完整回合结算 | O(V) | V ≤ 3N²+3N+1，N=5 时约 91 格 |

标准棋盘（半径 5）共 91 格，所有算法均为毫秒级，无性能问题。

### 13.2 关键边界情况

| 情况 | 处理方式 |
|------|---------|
| 落子后己方无气（自杀） | 规则未禁止，按现有逻辑：先移除敌方被吃群，再检查自身是否存活 |
| 落子同时触发吃子和圈地 | 严格顺序：先执行吃子（移除棋子），再执行领地结算 |
| 落子后领地变化（吃子打开缺口） | 吃子结算后重新执行领地算法，自动覆盖旧领地判断 |
| 棋盘已满无合法格 | `GameState` 在每回合开始检查，自动触发 `game_over` |
| 双方同时 Pass | `consecutive_passes >= 2` 即触发结束 |
| 角落/边缘格子邻格不足 6 | `get_neighbors` 仅返回棋盘内合法邻格，领地算法通过邻格数 < 6 检测边界接触 |

### 13.3 必要的单元测试清单

逻辑层必须覆盖以下测试用例（可用 GDScript 的 `assert` 编写独立测试场景）：

- [ ] Cube 坐标转像素，再反转回 Cube，结果一致
- [ ] Cube Round 在六边形边界上的取整正确性
- [ ] 半径 N 棋盘生成格子数 = 3N²+3N+1
- [ ] 单个棋子四面被围 → 气为 0 → 被吃
- [ ] 棋子群整体被围 → 整群被吃
- [ ] 落子后吃子，被吃格变为空格
- [ ] 完全封闭的空格区域（内无敌子）→ 形成领地
- [ ] 空格区域接触棋盘边界 → 不形成领地
- [ ] 空格区域内有敌方棋子 → 不形成领地
- [ ] 两方棋子混合围绕同一区域 → 不形成领地
- [ ] 计分：棋子数 + 领地格数求和正确
